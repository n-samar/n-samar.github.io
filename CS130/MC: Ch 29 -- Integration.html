McConnell: Chapter 29 — Integration
  • Integration refers to the software development activity in which you combine separate software components into a single system. 
  • The chronological sequence of construction clearly affects integration. You cannot integrate something that is not built.

Importance of the integration approach
  • It doesn’t matter if the building is only strong enough to stand once it is completed. It must be able to stand at every step of construction.
  • Constructing software in the wrong order makes it harder to code, harder to test, and harder to debug.
  • Benefits of careful integration:
    • Easier defect diagnosis
    • Fewer defects
    • Less scaffolding
    • Shorter time to first working product
    • Shorter overall development schedules
    • Better customer relations
    • Improved morale
    • Improved chance of project completion
    • More reliable schedule estimates
    • More accurate status reporting
    • Improved code quality
    • Less documentation
  • Phased integration
    • The norm until lately
    • well-defined steps:
      • Design, code, test, and debug each class. (unit development)
      • Combine the classes into one whopping-big system (system integration)
      • Test and debug the whole system (system dis-integration)
    • Issues: 
      • New problems inevitably arise upon integration and causes can be anywhere but the system is too big to debug properly.
      • All integration problems present themselves at once
      • Bugs can interact with bugs and cause new bugs
    • Might be good if program has 2 or 3 classes
  • Incremental Integration
    • Write and test a program in small pieces and then combine the pieces one at a time.
    • Follow these steps:
      • Develop small, functional part of system. Thoroughly test and debug it.
      • Design, code, test, and debug a class.
      • Integrate the new class with the skeleton. Test and debug the combination of skeleton and new class. If work remains return to step 2.
  • Advantages of incremental integration
    • Errors are easy to locate. When problem is involved in integration step, you know the new class is involved in it. You also reduce chance that multiple errors are interacting with each other in strange ways.
    • The system succeeds early in the project. When code is integrated and running, even if the system isn’t usable, it’s apparent that it soon will be. Programmers see early results from their work.
    • You get better progress monitoring. 
    • You will improve customer relations. Customers like signs of progress!
    • The units of the system are tested more fully. 
    • You can build the system with a shorter development schedule. If integration is planned carefully, you can design part of the system while another part is being coded.

Incremental integration strategies
  • Planning for integration defines the construction schedule
  • Top-down Integration
    • The class at the top of the hierarchy is written and integrated first. Stub classes are written at a lower level of abstraction. 
    • Stub classes get slowly replaced by implemented classes.
    • Interfaces between classes must be carefully specified
    • Most troublesome errors arise from poorly planned class interface compatibility.
    • Many disadvantages:
      • Tricky low level interfaces specified and implemented last
      • Performance issues usually arise at lower levels and cannot be predicted reliably in other levels of abstraction.
      • You need a bunch of stub classes very early on.
      • These stubs are more probable to contain errors because they often are not designed carefully.
    • If more practical, you can implement top-down through sections. This means you implement top section first and then divide code into sections and implement each section independently to completion before moving to next section. Many of top-down integration downsides still apply, but less obviously.
  • Bottom-Up Integration
    • Write and integrate low-level classes first.
    • Write test drivers to exercise the low-level classes initially and add classes to the test-driver scaffolding as they’re developed.
    • Restricts errors to the specific class being integrated
    • Easy to notice performance bottlenecks
    • If conceptual design problems present, we will not notice until we get to the top of the hierarchy, which is bad. If dramatic design changes necessary, lot of low-level work has to be discarded.
    • Also, letting low-level details drive the design of higher-level classes contradicts principles of information hiding and object oriented design.
    • As with top-down, you can divide system into sections and develop sections bottom-up one at a time.
  • Sandwich Integration
    • First integrate the high-level business-object classes at the top of the hierarchy.
    • Then integrate the device-interface classes and widely used utility classes at the bottom.
    • Leave middle level classes until later
  • Risk Oriented Integration
    • I.e. Hard Part First Integration
    • Coincidentally, similar to sandwich integration in most cases
    • Identify level of risk associated with each class
    • Experience suggests, top level classes are very risky
    • Also, low level classes are very risky
    • Also, some middle classes may turn out to be very challenging too 
  • Feature Oriented Integration
    • Integrate one feature at a time.
    • If you are writing a word processor, a feature may be displaying underlining on the screen or reformatting the document automatically — something like that.
    • Since each feature is self contained, each feature contains all support code it needs.
    • Each new integration step brings new and visible functionality.
    • Supports soliciting customer feedback
    • Works well with object-oriented design.
  • T-Shaped Integration
    • Select a full vertical slice and implement it first.

Daily Build and Smoke Tests
  • Reduces risk of low quality
  • Easier to diagnose defects
  • How to use/implement daily builds:
    • Build daily. Treat the daily build as the heartbeat of the project.
    • Check for broken builds. If daily build doesn’t work, fixing it becomes top priority. At minimum a good build should:
      • Compile all files, libraries and other components successfully.
      • Link all files, libraries and other components successfully.
      • Not contain any showstopper bugs that prevent the program from being launched or that make ti hazardous to operate. 
    • Smoke test daily. Smoke test should exercise entire system from end to end. Need not be exhaustive. Prevents against creeping integration issues.
    • Keep smoke test current. 
    • Automate the daily build and smoke test.
    • Establish a build group.
    • Add revisions to the build only when it makes sense to do so. Developers should commit to master only once they have a reasonable piece implemented correctly. Not everybody needs (or should) commit daily to each build.
    • Don’t wait too long to add a set of revisions. Break down construction of bigger feature or module into episodes that can be integrated so that you can benefit from daily integration.
    • Require developers to smoke test their code before committing.
    • Create staging area for code that is going to be added to the build. 
    • Create a penalty for breaking the build. If somebody breaks the build, they should stop all work until they fix it.
    • Release builds in the morning. This way testers can test on a fresh build that day. You also have better access to developers if errors are found.
    • Build and smoke test even under pressure. Under stress developers lose their discipline. They review and test their own code less carefully. In these situations, daily builds may be the only posts of sanity. Be sure to enforce them.
  • The larger the project the more important daily integration becomes!
