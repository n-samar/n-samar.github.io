McConnell: Chapter 22 — Developer Testing
  • Testing supported by industrial and academic research and by commercial experience
  • Unit testing. Test functions in isolation.
  • Component testing. Testing of whole classes.
  • Integration testing. Starts as soon as there are two classes to test and lasts until end of project.
  • Regression testing. Repetition of previously executed test cases.
  • System testing. The execution of software in its final configuration, including integration with other software or hardware.
  • Blackbox testing. Tester cannot see the inner workings of the item being tested.
  • White box testing. Tester is aware of inner workings.
  • Individual testing steps typically find less than 50% of errors present. All testing steps combined and less than 60% of errors.
  • Why is testing hard?
    • Successful tests break your code
    • Testing doesn’t prove absence of errors
    • Testing alone does not improve software quality. It’s like weighing yourself vs. losing weight.
    • If you execute the program hoping that it won’t have any errors, it will be too easy to overlook the errors you find. Keep in mind — better for you to find errors than someone else.
  • How much time to spend on testing?
    • Common answer: 50% of all developer time
    • BUT, this is misleading because it also includes debugging, independent testing time, and it states common practice not best practice.
    • Developer testing should take 8-25% of developer time.
  • Test thoroughly before you integrate.
  • Feeling of good test coverage doesn’t mean actual coverage is adequate.
  • Things you should test for:
    • Test for adherence to requirements. Plan for these tests during requirement gathering.
    • Test for each design concern
    • Use basis testing. At a minimum, test each line of code.
    • Keep a checklist of errors made on this project and all other projects.
  • Write tests first to minimize time between error introduction and error discovery. Other advantages of TDD:
    • Doesn’t take more time than the alternative
    • Detect/correct errors earlier
    • Forces you to think about requirements and design
    • Exposes requirements problems sooner
    • You can test both last and first
  • Limitations of developer testing:
    • Developers tend to test for whether the code works, instead of testing for all ways the code could break.
    • Immature testing organizations have 1 dirty test for every clean one. Mature organizations invert this number by introducing 25 times as many dirty tests.
    • Developer testing tends to instill optimistic view of test coverage.
    • Skips more sophisticated types of test coverage. 100% statement coverage is a good start, but far from adequate. 100% branch coverage is somewhat better.
  • Structured basis testing: cover all paths possible in program with minimum number of test cases.
  • Data-flow testing
    • Based on the idea that data usage is at least as error-prone as control-flow
    • Data exists in one of three states:
      • Defined. Data is initialized but not yet used.
      • Used.
      • Killed.
    • Another division:
      • Entered vs. Exited
    • Data should be ideally defined, then used a couple of times, then killed.
    • View following patterns suspiciously:
      • Defined-defined.
      • Defined-exited.
      • Defined-killed.
      • Entered-killed.
      • Entered-used.
      • Killed-killed.
      • Killed-used.
      • Used-defined.
    • Check for these anomalous state transitions before testing begins.
  • Equivalence partitioning. Partition data into equivalence classes that cause a specific branch of code to be executed. That way, you just test one data initialization per equivalence class and you can be sure you tested each data flow path.
  • Error guessing. Guess based on experience.
  • Good situations to test for
    • Boundary analysis. Off-by-one errors.
    • Compound boundaries. Set multiple objects to boundary values and see what happens.
    • Classes of bad data.
      • Too little data
      • Too much data
      • Wrong kind of data
      • Uninitialized data
    • Classes of good data. Nominal cases can have errors too:
      • Nominal data
      • Minimum normal configuration
      • Maximum normal configuration
      • Compatibility with old data
    • Use test cases that make hand-checks convenient. Use the most convenient data initialization for each equivalence class.

Other errors
  • Error correction often does not involve changing more than one routine.
  • Errors are often outside the construction domain. Most common error sources are thin app,action domain knowledge, changing requirements, communication.
  • Most construction errors are the programmer’s fault.
  • Clerical errors (typos) are a very common source of problems. (about 36% of construction errors)
  • Misunderstanding the design is frequent. Understand the design is worth its time.
  • Most errors are easy to fix (28.5% can be fixed with less than 1 hour).
  • Do requirements and design reviews frequently upstream.
  • Measure error-related data in your organization. Error principles don’t generalize well between organizations.
  • On small projects, construction errors are most common (~75%)
  • Construction errors account for 35% of all errors regardless of project size.
  • How manny errors should you expect to find?
    • Industry average is 1-25 errors per 1 kLOC.
    • App Division of Microsoft: 10-20 errors per 1 kLOC during in-house testing; 0.5 defects per 1 kLOC in released product.
    • TSP + cleanroom significantly reduce these numbers; train developers not to make error in the first place.
  • Errors in testing itself:
    • Check your work. Step through test cases with debugger.
    • Keep your test cases.
    • Plug unit tests into a testing framework. This prevents the tendency to throw test cases away.
  • Test-support tools:
    • Build scaffolding to test individual classes. This means using mock objects. Possible mock approaches:
      • Return control immediately, with no action taken
      • Test the data fed to the mock routine/object
      • Print diagnostic message or log a message
      • Get return values from interactive input
      • Return standard answer regardless of input
      • Burn up the number of clock cycles that the real routine would need to execute
      • Function as a slower, duper version of actual routine
    • Another scaffolding approach is writing fake routines that calls the real routine being tested. These are called drivers or test harnesses. This type of routine can:
      • Call the object with a fixed set of inputs
      • Prompt for input interactively and call object with it
      • Take command line arguments and call the object
      • Read arguments from a file and call the object
      • Run through predefined sets of input through multiple calls to the object
      • Dummy files
  • Regression testing is a lot easier if you have automated tools to check the actual against expected output.
  • Test data generators
    • Properly designed random data generators can generate unusual combinations of test data that you wouldn’t think of
    • Random data generators can exercise your program more thoroughly than you can
    • Modular design pays off during testing
    • You can reuse a test driver if the code it tests ever has to be changed.
  • Data Recorder/Logging
    • Record the system state prior to an error and detail the exact error conditions.
  • Symbolic debuggers
    • Less labor intensive than formal code review.
  • System perturbers
    • Memory filling. You want to be sure you don’t have any uninitialized variables. Fill memory when it is freed, e.g. insert breakpoint interrupts into freed memory.
    • Memory shaking. In multitasking systems, some tools can rearrange memory as your program operates so that you can be sure you haven’t written any code that depends on data being in absolute rather than relative locations.
    • Selective memory failing. A memory driver can simulate low-memory conditions in which a program might be running out of memory, fail on a memory request, grant an arbitrary number of memory requests before failing, or fail on an arbitrary number of requests before granting one. This is especially useful for testing complicated programs that work with dynamically allocated memory.
    • Memory-access checking (bounds checking). 
  • Error databases
    • Both a management and a technical tool.

Improving Your Testing
  • Have to know exactly what you are doing so that you can vary it slightly and observe the effects of the variation.
  • When you observer a change that has a positive effect, you modify the process so that it becomes a little better.
  • Planning to test. Important, because testing methods need to be repeatable. Important, because inadequacies in quality have to be matched to specific bad practices.
  • Automated testing. 
    • Estimates say that about only half of all manual tests are executed correctly.
    • An automated test has a lower chance of being wrong.
    • Automated tests can be run frequently.
    • Help detect errors earlier rather than later
    • Used as safety net for securing functional builds
    • Especially useful in new, volatile technology environments.

Keeping Test Records
  • Aside from making the testing process repeatable, you need to measure the project so that you can tell for sure whether changes improve or degenerate it. Here are a few kinds of data you can collect to measure your project:
    • Administrative description of the defect (the data reported, the person who reported it, a title or description, the build number, the date fixed)
    • Full description of the problem
    • Steps to take to repeat the problem
    • Suggested workaround for the problem
    • Related defects
    • Severity of the problem — for example, fatal, bothersome, or cosmetic
    • Origin of the defect: requirements, design, coding or testing
    • Subclassifications of a coding defect: off-by-one, bad assignment, bad array index, bad routine call, and so on
    • Classes and routines changed by the fix
    • Number of lines of code affected by the defect
    • Hours to find the defect
    • Hours to fix the defect
  • Using bug records for software quality metrics:
    • Number of defects in each class, sorted from worst class to best, possibly normalized by class size
    • Number of defects in each routine, sourced from worst routine to best, possibly normalized by routine size
    • Average number of testing hours per defect found
    • Average number of defects found per test case
    • Average number of programming hours per defect fixed
    • Percentage of code covered by test cases
    • Number of outstanding defects in each severity classification
