Lecture Notes: February 28 — Dependability and Security
  • Dependability: You want the SW to work even under adverse real-world conditions
  • Security: You want the SW to work even under the presence of adversaries.
  • Security and dependability are often more important than functionality
    • Failures could cost more than your entire system
    • Bad PR
  • Security and dependability are system aspects, i.e. security and dependability are emergent
    • You can’t write a dependability module
  • Must also consider:
    • Failures in other modules
    • Failures in the hardware, networking
    • Operational failures
  • Parameters of dependability:
    • security (although this is often separate)
    • safety (damage to user/environment)
    • reliability (how likely is it to operate correctly?)
    • availability (how likely is it to be up and running?)
    • resilience (how well does the system recover from failures?)
    • maintainability
    • repairability
    • error tolerance (user mistakes do not cause failures)
  • Terminology:
    • human error — mistake by the designer
    • fault — latent problem in the system (static property)
    • system error — bad internal state of the system
    • failure — observable bad behavior
  • Safety terminology:
    • accident — unplanned event that causes injury
    • hazard — a condition that might lead to an accident
    • damage — cost measure for accidents
    • hazard severity — worst damage expected from hazard
    • risk — probability of an accident
  • Security terminology:
    • exploit — unwanted event causing something bad
    • vulnerability — condition that might lead to an exploit
    • exposure — system is available and exposed to attackers
    • attack — an attempt to exploit the system
    • threat — vulnerability that might be attacked
    • control — mechanism to defang a vulnerability
  • Dependability engineering components:
    • Programming
    • Architectures
    • Software processes
    • Requirements
  • Specifying dependability with requirements:
    • often the hardest
    • easy to think that you are done
    • tempting to over specify
  • Basic approach
    • Identify risks
    • Analyze risks (prioritize)
    • Decompose the risks into root causes
    • Reduce risks
  • How to deal with faults?
    • avoid — e.g., avoid out of bound index by not using arrays
    • Detect and remove — e.g., subscript checking or exception handling
    • Tolerate the fault — e.g., a[i] == 0 if i is out of bounds
  • Dependable software processes
    • Used to convince the customer that our product is dependable
    • Includes paperwork
      • test plan + management
      • static analysis
      • formal inspection
      • formal specification
      • requirements reviews
      • system modeling
    • You want:
      • something that is audible
      • something that is redundant
      • something that is robust
  • Dependable programming practices:
    • information hiding
    • input validity checking
    • exception handling
    • timeouts
    • checkpoint restart
  • Omit the tricky stuff:
    • gotos
    • floating point
    • pointers
    • subscripts
    • recursion
    • new
    • parallelism
    • traps/interrupts
    • inheritance
    • aliasing
Security
  • Why it’s harder than dependability?
    • Attacks are not random
    • Attackers try to obscure themselves
    • DoS
  • Security design guidelines:
    • Have an explicit security policy and use it
    • Defense in depth
    • Fail securely
    • Balance security and usability
    • Log user actions
    • Use redundancy and diversity to reduce risk
    • Specify input formats
    • Compartmentalize assets
    • Design the deployment
    • Design the recovery
  • Security architecture guidelines:
    • Don’t try to retrofit security
    • Fallbacks should be more secure than the original
    • No single point of failure
    • Keep it understandable
