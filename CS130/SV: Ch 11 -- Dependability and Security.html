Sommerville: Chapter 11 — Dependability and security
  • Dependability of systems is now usually more important than their detailed  functionality because:
    • System failures affect a large number of people.
    • Users often reject systems that are unreliable, unsafe, or insecure. 
    • System failure costs may be huge. 
    • Undependable systems may cause information loss. Data is often worth much more than the system on top of which it is processed.
  • For dependability, consider the following during design (dependability design):
    • Hardware failure. System hardware may fail because of mistakes in its design.
    • Software failure.
    • Operational failure. Human users may fail to use or operate the system correctly.
  • The dependability of a computer system is a property of the system that reflects its trustworthiness. 
  • Four principle properties of dependable systems:
    • Availability. The probability the system will be up and running and able to deliver useful services.
    • Reliability. The probability that the system will correctly deliver services as expected by the user.
    • Safety. Probability that the system will cause damage to people or the environment.
    • Security. How well can the system resist accidental or deliberate intrusions?
  • To develop dependable software, ensure that:
    • You avoid the introduction of accidental errors into the system during software specification and development.
    • You design verification and validation processes that are effective in discovering residual errors that affect the dependability of the system.
    • You design protection mechanisms that guard against external attacks that can compromise the availability or security of the system.
    • You configure the deployed system and its supporting software correctly for its operating environment.
    • Include recovery mechanisms.

Availability and reliability
  • If you measure system reliability in one environment, you can’t assume that the reliability will be the same if the system is used in a different way.
  • Failure/Fault/Error:
    • Failure. An event that occurs at some point in time when the system does not deliver a service as expected by the user.
    • Error. An erroneous system state that can lead to system behavior that is unexpected by system users.
    • Fault. A characteristic of a software system that can lead to system error.
  • System faults do not always result in system errors and system errors do not necessarily result in system failures. Why?
    • Not all code in a program is executed.
    • Errors are transient. 
    • The system may include fault detection and protection mechanisms.
    • Users adapt their behavior to avoid failures.
  • Three complementary approaches that are used to improve the reliability of a system:
    • Fault avoidance. Avoiding error-prone programming language constructs such as pointers and use of static analysis to detect program anomalies.
    • Fault detection and removal. The use of verification and validation techniques that increase the chances that faults will be detected and removed before the system is used.
    • Fault tolerance. Techniques that ensure that faults in a system do not result in system errors or that system errors do not result in system failures.

Safety
  • Two classes of safety-critical software:
    • Primary safety-critical software. Software that is embedded as a controller in a system. Malfunctioning of such software can cause a hardware malfunction.
    • Secondary safety-critical software. Software that can indirectly result in an injury. E.g. faulty CAD system.
  • How come reliable system need not be necessarily safe?
    • Undetected faults can be dormant for a long time and software failures can occur after many years of reliable operation.
    • The specification may be incomplete in that it does not describe the required behavior of the system in some critical situation.
    • Hardware may break and put the system in an unanticipated, unsafe state.
    • The system operators may generate inputs that are not individually incorrect but which can lead to a system malfunction in some situation.
  • Safety terminology:
    • Accident. An unplanned event or sequence of events which results in human injury, damage of property or the environment.
    • Hazard. A condition with the potential for causing for contributing to an accident.
    • Damage. A measure of the loss resulting from a mishap.
    • Hazard severity. An assessment of the worst possible damage that could result from a particular hazard.
    • Hazard probability.
    • Risk. This is a measure of the probability that the system will cause an accident.
  • Safety management involves insuring the accident does not occur and minimizing consequences of accidents. This can be done through:
    • Hazard avoidance. System is designed so that hazards are avoided.
    • Hazard detection and removal. The system is designed so that hazards are detected and removed before they result in an accident.
    • Damage limitation. The system may include protection features that minimize the damage that may result from an accident.
  • Analysis of accidents indicates they were almost all due to a combination of failures in different parts of a system. Research suggests there are no good ways to anticipate all possible combinations of failures.

Security
  • If you really want a secure system, it is best not to connect it to the Internet.
  • In network systems, there are three main types of security threats:
    • Threats to the confidentiality of the system and its data. 
    • Threats to the integrity of the system and its data.
    • Threats to availability of the system and its data.
  • Controls that can be put in place to increase system security:
    • Vulnerability avoidance. 
    • Attack detection and neutralization. Controls intended to detect and repel attacks.
    • Exposure limitation and recovery.
