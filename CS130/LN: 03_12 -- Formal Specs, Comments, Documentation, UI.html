Lecture Notes: March 12 — Formal Specs, Comments, Documentation, UI
Formal Specs
  • behavioral specs (invariants, preconditions, postconditions)
  • Z language
    • signature
    • predicates
  • e.g. of Z description
    • BlockHandler
      • singature
        • used, free: Power set of Blocks
        • BlockQueue: sequence of power set of Blocks
      • invariants
        • used CROSS free = empty set
        • used UNION free = AllBlocks
        • For all i in the domain of BlockQueue : BlockQueue(i) is subset of used
        • Fot all i, j in the domain of BlockQueue : if i is not equal to j then BlockQueue(i) \ BlockQueue j = empty set (i.e. a BlockQueue cannot contain the same block in two different indices)
    • RemoveBlocks
      • signature
        • Delta BlockHandler
      • invariants
        • # BlockQueue > 0 (i.e. the number of entries in BlockQueue is greater than 0)
        • used' = used \ head BlockQueue (i.e. the remove function’s specification; used’ means new value of used)
        • free’ = free UNION head BlockQueue
        • BlockQueue’ = tail BlockQueue
  • When you implement, you want to make sure that the invariant holds after all operation. That is, we want to prove INV AND RemoveBlocks IMPLIES INV’
  • Downsides of formal methods:
    • They are hard
    • easy to over formalize
    • not a substitute for understanding
    • they still need comments
    • they don’t guarantee that your code is bug free
    • they’re like software, need engineering
  • Fancy static analysis (Coverity):
    • Specify invariant that you want to hold
    • Coverity runs through all the code and makes sure the invariant holds
  • Proof-carrying code

Comments and Documentation
  • Overall goal of both: to save work
  • Problem: you save work later by doing it now (delayed gratification)
  • Large projects need more documentation due to
    • internal communication
    • user communication

Comments
  • How to avoid comments?
    • Your code itself can be commentary (naming conventions are commentary)
    • Group related items together
    • Use a consistent level of abstraction/refactor
  • When to comment?
    • Put yourself in the reader’s shoes (pretend you haven’t seen the code before)
  • Problems
    • Missing comments
    • Redundant comments
    • Misleading or wrong comments
  • How many comments? 10% of the lines, says McConnell 
  • Don’t waste your time formatting
  • Keep them close to the code that they describe
  • Treat them like code (if the comment is important enough to write down. it is valuable enough to maintain)
  • Tool: Javadoc
    • Allows literate programming (single document contains commentary and code)

Documentation
Process documentation
  • Documenting the procedures you use to build the software
  • Arguably the most important (you must describe your build procedure and code structure to build new developers up to speed)
  • Types of process documentation:
    • Traditionally done through email instructions
    • Working papers: goal is to describe the rationale and details of the chosen design
    • Plan
    • Coding standards
  • Shouldn’t be in code comments
  • Without this you get lost
  • All of this stuff should be under Version Control
  • Contracts and regulatory requirements
Product documentation
  • consider the audience:
    • novices (tutorial)
    • expert users (reference manual)
    • operations staff
    • purchasers, reviewers, managers
    • developers
  • IEEE Standard 26514-2010
    • Title Page with ID string that uniquely identifies it
    • Chapters with structure
    • Glossary
    • Index
    • Troubleshooting (FAQ)

User Interface
User Interface Design (User-Centered Design)
  • A subset of design
  • Harder to test (you need real users)
  • Users are hard to predict
  • crucial: learn how the users will behave
  • Google’s material design
  • General principles part I:
    • Consistency matters
    • Compatibility with existing practice
    • Use real-world metaphor
    • Disclose information progressively (managing information flow to user)
    • Make context explicit
    • Provide reasonable defaults
    • Provide shortcuts
    • Overall goal: Minimize the memory load on the user
  • General principles part II:
    • Put the user in control
    • No dead ends
    • Flexible interaction (undo/redo)
    • Customizable (for power users)
  • Common issues:
    • Response time (defined in spec)
    • Help
    • Error handling
    • Shortcuts/commands vs. mouse/menu vs. touchscreen/gesture
    • Accessibility
    • Internationalization

Web UI issues
  • Mask latency issues
    • Anticipate the next request (preload guessed answers)
    • Pipelining requests (send several requests at the same time)
  • You need to keep track of state and save work (who should track state — client or server?)
  • Cookies
  • Same UI shared among many apps?
    • run all apps in a single browser: adds consistency, reduces flexibility

Mobile App UI design issues
  • metaphors matter more
  • direct manipulation
  • typing is more of a pain
  • immediate feedback is even more crucial because of error rates (error rates are higher on cellphones)
  • aesthetics matter more
  • small screen

Accessibility
  • UI’s for people with disabilities
  • Guidelines:
    • provide text alternative: <img href=“…jpg” alt=“UNDO button”>
    • synchronize multimedia alternatives (captions)
    • don’t lose useful information in the alternative
    • make all functionality useful from a keyboard
    • give users enough time
    • don’t create content known to cause seizure (>3 flashes per second)
