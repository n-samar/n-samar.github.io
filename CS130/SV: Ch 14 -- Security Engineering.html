Sommerville: Chapter 14 — Security Engineering
  • Topic: What issues should be considered when designing secure application systems?
  • When you consider security issues, you have to consider both the application software and the infrastructure on which this system is built.
  • The infrastructure for complex applications may include:
    • an OS
    • Other generic applications like web browsers and e-mail clients
    • Database management systems
    • Middleware
    • Libraries of reusable components
  • Important distinction between application and infrastructure security:
    • Application security is a software engineering problem.
    • Infrastructure security is a management problem.
  • Activities related to system security management:
    • User and permission management includes adding and removing esters from the system, ensuring that appropriate user authentication mechanisms are in place.
    • System software deployment and maintenance includes installing system software and middleware and configuring these properly so that security vulnerabilities are avoided.
    • Attack monitoring, detection and recovery. 

Security risk management 
  • Concerned with assessing the possible losses that might ensue from attacks on assets in the system, and balancing these losses against the costs of security procedure that may reduce these losses.
  • Risk assessment stages:
    • Preliminary risk assessment. Decisions on the detailed system requirements, the system design, or the implementation technology have not been made. The aim is to decide if an adequate level of security can be achieved at a reasonable cost. Focus on security requirements.
    • Life-cycle risk assessment. Takes place during the system development life cycle and is informed by the technical system design and implementation decisions. 
    • Operational risk assessment. After a system has been deployed and put into use, risk assessment should continue.
  • Types of security threats:
    • Interception threats that allow an attacker to gain access to an asset.
    • Interruption threats that allow an attacker to make part of the system unavailable. E.g. denial of service attack.
    • Modification threats that allow an attacker to tamper with a system asset.
    • Fabrication threats that allow an attacker to insert false information into a system.

  • Lifecycle risk assessment
    • Identifies the design and implementation details that affect security. 
    • It may affect future design decisions.

Design for security
  • Generally, it is very difficult to add security to a system after it has been implemented.
  • Application-independent issues relevant to secure systems design:
    • Architectural design — how do architectural design decisions affect the security of the system?
    • Good practice — what is accepted good practice when designing secure systems?
    • Design for deployment — what support should be designed into systems to avoid the introduction of vulnerabilities when a system is deployed for use?
  • Tension between security and usability/performance.

Architecture Design
  • Two fundamental issues in designing for security:
    • Protection — how should the system be organized so that critical assets can be protected against external attack?
    • Distribution — how should system assets be distributed so that the effects of a successful attack are minimized?
  • Protection vs. Distribution: So, you can build your protection around the whole system and therefore build a very strong protection shield. However, if the protection is successfully attack, the whole system is exposed. For a well distributed approach, the inverse is true.
  • Layered approach: The number of protection layers that you need in any particular application depends on the criticality of the data.
  • If protection of data is critical, then a client-server architecture should be used. However
    • Servers are vulnerable to DoS attacks.
    • Costs of attacks are high, as are repair costs
  • If DoS attacks are a serious concern, you may decide to use a distributed object architecture for the application. Here, data replication across nodes could be used.
  • Architecture style best suited to meeting security requirements may not be the best approach for the performance requirements.

Design Guidelines
  • Main uses of general design guidelines:
    • They help raise awareness of security issues in a software engineering team.
    • They can be used as a review checklist that can be used in the system validation process.
  • General principles not directly related to software engineering:
    • Keep It Simple
    • Avoid security through obscurity
    • Secure the weakest link
  • Base security decisions on an explicit security policy.
  • Avoid a single point of failure. A single failure in part of the system should not result in an overall systems failure.
    • Do not rely on a single method to ensure security. Instead do ‘defense in depth’.
  • Fail securely.
  • Balance security and usability.
  • Log user action. Who did what, assets used, and the time and date of the action.
    • Build tools to scan the logs and discover attack attempts.
  • Use redundancy and diversity to reduce risk.
  • Validate all inputs. A common attack on a system involves providing the system with unexpected inputs that cause it to behave in unanticipated ways.
    • You should never accept any input before applying some checks to it.
  • Compartmentalize your assets.
    • Compartmentalize = Do not provide all-or-nothing access to information in a system.
  • Design for deployment
    • Many security problems arise because the system is not configured correctly when it is deployed in its operational environment.
    • Always design your system so that facilities are included to simplify deployment in the customer’s environment and to cake for potential configuration errors and omissions in the deployed system.
  • Design for recoverability. Think about how to recover from possible failures and restore the system to a secure operational state.

Design for deployment
  • The deployment of a system involves configuring the software to operate in an operational environment, installing the system on the computers in that environment, and then configuring the installed system for these computers.
  • Software designers have the responsibility to design for deployment. 
  • Four ways to incorporate deployment support in a system:
    • Include support for viewing and analyzing configurations. 
      • Allow administrators or permitted users to examine the current configuration of the system. This makes it easier to sport errors and omissions.
    • Minimize default privileges.
      • You should design software to that the default configuration of a system provides minimum essential privileges.
    • Localize configuration settings.
      • Ensure that everything in a configuration that affects the same part of a system is set up in the same place.
    • Provide easy ways to fix security vulnerabilities. 
      • You should include straightforward mechanisms for updating the system to repair security vulnerabilities that have been discovered.
      • Provide an easy way for administrators to install updates.

System survivability
  • Survivability and resilience is an emergent property of a system as a whole, rather than property of individual components, which may not themselves be survivable. 
  • Maintaining the availability of critical services is the essence of survivability.
  • You need to specify:
    • the system services most critical for a business
    • the minimal quality of service that must be maintained
    • how these services might be compromised
    • how these services can be protected
    • how you can recover quickly if the services become unavailable.
  • Three complementary strategies for achieving survivability:
    • Resistance. Avoiding problems by building capabilities into the system to repel attacks.
    • Recognition. Using mechanisms for recognizing attacks.
    • Recovery. Tolerating problems by building capabilities into the system to deliver essential services while under attack.
  • Stages of survivable systems analysis:
    • System understanding. Review the goals of the system, the system requirements, and the system architecture.
    • Critical service identification. The services that must always be maintained and the components that are required to maintain these services are identified.
    • Attack simulation. Scenarios or use cases for possible attacks are identified along with the system components that would be affected by these attacks.
    • Survivability analysis. Components that are both essential and comprisable by an attack are identified and survivability strategies based on resistance, recognition, and recover are identified.
