Lecture Notes: February 21 — Project Management
  • White box testing
  • Black box testing advantages
    • Modularizes tests from code
    • Test before code
    • Test group can be independent check
  • Testing ideas:
    • Scenario-based tests
    • Fault-based testing: focus on areas most likely to break
    • Inheritance-based testing: child must pass all parents’ tests
    • Random testing: not great by itself, but cheap
  • Problems with testing
    • tests are often too closely coupled to the code (black boxing tries to address this)
    • tests are never comprehensive (combinatorial explosion)
    • some things are too big of a pain to test (anything non-deterministic, e.g. UI)
  • How much budget to allocate for testing?
    • Wait for the money to run out
    • Wait for time to run out
    • Use metrics: if we don’t have bugs historically, maybe test less; if lots of bugs, then test more.

System building basics
  • How to get end application from source elements?
    • Source code —> System building —> Application
  • Target, development, build machines
  • Cross-compiler: runs on X, but generates code for Y.
  • Toolchains
  • Automation is key
    • Shell scripts connected by subshells
  • So, I can connect shell scripts to make my toolchain? Not efficient enough.
  • One abstraction up: Make — allows for:
    • partial compilation
    • resume partially failed script
  • Is make higher level enough?
    • No, we have configure — generates files that allow make to run; one level of automation above make.
  • Autoconf: one level of abstraction above configure
  • Automake: one level of abstraction above autoconf.
  • Software Configuration Management (SCM) questions:
    • What are the software configuration items (SCIs)? (identification)
    • How do you record different versions? (version control)
      • A user is reporting an error and wants to report it, how can he find out which version he is using? Git cannot solve this, and git does not address all issues of version control.
    • How do you control changes? (change control)
      • Who is responsible for approving changes?
    • How do you ensure changes are done right? (configuration auditing)
    • How do you notify others of change? (reporting)
      • You have to report selectively in order not to swamp people with stuff they don’t want to know.
    • How to generate system from sources? (building)
    • How to get it into people’s machines? (release)
  • Baseline: includes spec, requirements, code, test cases that are reviewed, approved and known to work.
  • We build on the baseline to create new software.
  • You can always fall back to your baseline.
  • Optimistic version control: Anyone can work on anything (git)
  • Pessimistic version control: Only one developer allowed to work on each file
  • Standalone version control: Repository is on one system. Need to be logged into that system to make changes.
  • Client-server version control: Repository on server, developers work on clients (CVS)
  • Distributed version control: Repositories all over the place and are allowed to diverge. 
    • Tools provided to fix inconsistencies
    • Semiautomated merging
    • git
  • Features of Version Control:
    • Isolation: different developers can work on different versions without stepping on each other’s toes.
    • Selective undo
    • Update
    • Logs
    • Backups
  • TODO lists
    • Bugzilla — bug database
    • Estimate cost of fixes
    • Watch out for big bug reports (big feature requests and scope creep)
  • Change Control Board

  • How NOT to manage projects. Danger signs [Reel 1999]:
    • Managers and developers avoid best practices (you know what you need to do, but you don’t do it)
    • Sponsorship is lost (or was never there)
    • Project lacks people with the right skills
    • Users don’t want the system
    • The business needs change
    • Unrealistic deadlines
    • Chosen technology changes
    • Changes are managed poorly
    • Product scope is not defined
    • Developers don’t understand the customer’s needs
  • 2+ danger signs —> get worried
  • 3+ danger signs —> leave the project
  • Which of the above are software engineering issues (vs. business issues)?
    • Unrealistic deadlines
    • Chosen technology changes
    • Changes are managed poorly
    • Product scope isn’t defined
    • Developers don’t understand customer needs
