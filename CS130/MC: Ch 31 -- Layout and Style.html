McConnell: Chapter 31 — Layout and Style
  • This chapter turns to an aesthetic aspect of computer programming: the layout of program source code.
  • Fundamental Theorem of Formatting: Good visual layout shows the logical structure of a program
  • The specific layout used inside the project is less important tan the consistency of whichever style is chosen.

Objectives of good layout
  • A good layout schemeshould do the following
    • Accurately represent the logical structure of the code.
    • Consistently represent the logical structure of code.
    • Improve readability.
    • Withstand modifications. Modifying one line of code shouldn’t require modifying several others.
  • What can white space be used for?
    • Grouping. Use white space to group related statements together.
    • Blank lines.The start of a new paragraph of code should be identified with a blank line.
      • Optimal number of blank lines is between 8 and 16 percent. If above 16 percent, debug time goes up dramatically.
    • Indentation. You should indent statements under the statement to which they are logically subordinate.
      • Two to four space indentation is optimal.

  • Laying out blocks
    • Pure blocks.
                        if (pixelColor = Red) {
                                s1;
                                s2;
                        }
    • Using begin-end pairs to designate block boundaries.
                        if (pixelColor = Red) 
                        {
                                s1;
                                s2;
                        }


  • Format single statement blocks consistently
  • For complicated expressions, put separate conditions on separate lines.
  • Avoid gotos.

  • Individual Statement Layout
    • Use spaces for expression clarity. They hardly ever hurt. 
    • The 80-line limit is now increasingly arbitrary.
    • Use spaces within parenthesis.
    • Use statements to make array references readable.
    • Use spaces to make routine arguments readable.

  • Formating Continuation Lines
    • One of the most vexing problems of program layout is deciding what to do with the part of a statement that spills over to the next line.
    • Make the incompleteness of the statement obvious. Make the part in the first line of the expression look blatantly incorrect if it were alone.
    • Keep closely related elements together.
    • Indent routine-call continuation lines the standard amount.
    • Make it easy to find the end of a continuation line.
    • Indent control-statement continuation lines the standard amount.
    • Do not align right sides of assignment statements.
    • Indent assignment-statement continuation lines the standard amount.

  • Use only one statement per line
  • In C++, avoid using multiple operations per line.

  • Laying out data declarations
    • Use only one data declaration per line.
    • Declare variables close to where they are first used.
    • Order declarations sensibly. E.g. group them by type.
    • In C++ put the asterisk next to the variable name in pointer declarations or declare pointer types.

  • Laying out comments
    • Indent a comment with its corresponding code.
    • Set off each comment with at least one blank line.

  • Laying out routines
    • Use blank lines to separate parts of a routine.
    • Use standard indentation for routine arguments.

  • Laying out classes
    • Laying out class interfaces:
      • Header comment that describes the class and provides any notes about the overall usage of the class
      • Constructors and destructors
      • Public routines
      • Protected routines
      • Private routines and data members
    • Laying out class implementations:
      • Header comment that describes the contents of the file the class sis in
      • Class data
      • Public routines
      • Protected routines 
      • Private routines
  • If you have more than one class in a file, identify each class clearly.

  • Laying out files and programs
    • Put one class in one file
    • Give the file a name related to the class name
    • Separate routines within a file clearly. This means at least two blank lines between routines.
    • Sequence routines alphabetically.
    • In C++ order the file like this:
      • File-description comment
      • #include files
      • Constant definitions that apply to more than one class (if more than one class in file)
      • Enums that apply to more than one class (if more than one class in file)
      • Macro function definitions
      • Type definitions that apply to more than one class (if more than one class in file)
      • Global variables and functions imported
      • Global variables and functions exported
      • Variables and functions that are private to the file
      • Classes, including constant definitions, enums, and type definitions within each class
