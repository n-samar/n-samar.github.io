Sommerville: Chapter 23 — Project Planning
  • As a manager, you have to break down the work into parts and assign these to project team members, anticipate possible problems and prepare tentative solutions to these problems.
  • Project planning through three different stages of project lifecycle.
    • At the proposal stage
      • Does your team have the necessary resources to complete the work?
      • What price should we at?
    • During the project startup phase 
      • How to break down project into increments?
      • How will resources be allocated?
      • Refine initial estimates from the first step
    • Plan modifications throughout project development
      • Due to new knowledge, emergent properties, or changing requirements.
  • Three main parameters to be used when computing the costs of a software development project:
    • Effort costs (the costs of paying software engineers and managers)
      • Usually the biggest cost.
    • Hardware and software costs, including maintenance
    • Travel and training costs
  • The price of a software product to a customer is the cost of development plus profit for the developer.

  • What should a plan contain:
    • Introduction. Describes objective of the projects and sets out constraints.
    • Project organization. Describes how the development team is organized. 
    • Risk analysis. 
    • Hardware and software resource requirements. Including delivery schedules.
    • Work breakdown. Identifies milestones, activities , and the allocation of people to activities.
    • Monitoring and reporting mechanisms. Defines the management reports that should be produced.

The Planning Process
  • Iterative 
  • Milestones are points in the schedule against which you can assess progress.
  • Deliverables are work products that are delivered to the customer
  • If there are serious problems with the development work that are likely to lead to significant delays, you need to initiate risk mitigation actions to reduce the risks of project failure.

Project Scheduling
  • Define tasks, estimate time needed to complete each task, the effort required, and who will work on the tasks that are identified.
  • Tasks should normally last at least a week, and no longer than 2 months.
  • The maximum amount of time for any task should be around 8 to 10 weeks. If it takes longer than this, the task should be subdivided.
  • You must take into account the possibility that things will go wrong.
  • You might want to fudge your estimates by 30-50%
  • Schedule representation:
    • Schedules can be maintained in a table or spreadsheets.
    • However this style of representation makes it difficult to see the relationship and dependencies between different activities.
    • For this reason, graphical representations of schedules are often used.
    • Two types of representations:
      • Bar charts, calendar based, show who is responsible for each activity, the expected elapsed time, and when the activity is scheduled to begin and end.
      • Activity networks, which are network diagrams, show the dependencies between the different activities making up a project.
    • Project activities are basic planning elements. Each activity has:
      • A duration in calendar days or months
      • An effort estimate, which reflects the number of person-days or person-months to complete the work
      • A deadline by which the activity should be completed
      • A defined endpoint. This represents the tangible result of completing the activity.
    • Each milestone should be documented by a short report that summarizes the progress made and the work done.
    • Delays can cause serious problems with staff allocation, especially when people are working on several projects at the same time, especially if there is a limited number of people that can work on the tasks.
    • In agile methods, we have release and iteration planning:
      • Release planning: looks ahead for several months and decides on the features that should be included in a release of a system
      • Iteration planning: has a shorter-term outlook, and focuses on planning the next increment of a system.
    • Two upsides to story-based agile task allocation:
      • The whole team gets an overview of the tasks to be completed in an iteration.
      • Individual developers choose the tasks to implement. This adds a sense of ownership.
    • If the work cannot be completed in the time allowed, the XP philosophy is to reduce the scope of the work rather than extend the schedule. 

Estimation Techniques
  • Project estimates are often self-fulfilling.
  • Two techniques:
    • Experience based techniques.
    • Algorithmic cost modeling. 
  • Usually if effort estimate is X, it will end up being somewhere between 0.25X and 4X.
  • The size and complexity of the system often incurs exponential development costs.
  • Algorithmic models should be appropriately calibrated to account of organization specific trends.
  • Algorithmic modeling should always estimate the worst, expected, and best costs.
  • The COCOMO II Model
    • Takes into account modern trends
    • Submodes that COCOMO II is made up of:
      • An application-composition model. This models the effort required to develop systems that are created from reusable components, scripting, or database programming.
      • An early design model. This model is used during early stages of the system design after the requirements have been established.
      • A reuse model. This model is used to compute the effort required to integrate reusable components and/or automatically generated program code.
      • A post-architecture model. Once the system architecture has been designed, a more accurate estimate of the software size can be made.

The application-composition model
  • Introduced into COCOMO II to support the estimation of effort required for prototyping projects and for projects where the software is developed by composing existing components.
  • PM = (NAP * (1 - %reuse/100))/PROD
    • PM — effort estimate in person-months
    • NAP — total number of application points in the delivered system.
    • % reuse — estimate of the amount of reused code in the development.
    • PROD — Application-point productivity

The early design model
  • This model may be used during the early stages of a project, before detailed architecture design specifications are available.
  • Both application-composition model and early design model assume zero reusable code integration cost
  • Effort = A * Size^B * M
    • Size — KSLOC
    • A = 2.94
    • 1.1 < B 1.24
    • M = PERS * RCPX * RUSE * PDIF * PREX * FCIL * SCED
      • RCPX — Product Reliability and Complexity
      • RUSE — Reuse Required
      • PDIF — Platform Difficulty
      • PERS — Personnel capability
      • PREX — Personnel experience
      • SCED — Schedule
      • FCIL — Support Facilities
      • These values estimated by integers between 1 and 6.

The reuse model
  • A model used to estimate the effort required to integrate reusable or generated code.
  • PM_auto = (ASLOC * AT / 100) / ATPROD
    • ASLOC — total number of lines of reused code, including code that is automatically generated.
    • AT — the percentage of reused code that is automatically generated
    • ATPROD — productivity of engineers in integrating such code
    • ATPROD is ~2,400 source statements per month.
  • ESLOC = ASLOC * AAM
    • ESLOC — equivalent number of lines of new source code
    • ASLOC — number of lines of code in the components that have to be changed
    • AAM — Adaptation Adjustment Multiplier. AAM adjusts the effort estimate of changing code to reflect the additional effort required to changing vs. initially building code.
  • AAM = AAF + SU + AA
    • AAF — Adaptation component that represents the costs of making changes to the reused code.
    • SU — Understanding component that represents the costs of understanding the code to be reused and the familiarity of the engineer with the code.
    • AA — Assessment factor that represents the costs of reuse decision making. It takes time to decide wether a component is adequate for reuse.
  • ESLOC = ASLOC * (1 - AT / 100) * AAM — Assuming some part of the code can be automatically generated (AT).

The post-architecture level
  • PM = A * Size^B * M (Same as early design estimates)
  • By this stage, your predictions should be more accurate.
  • Estimating code size should be based on 2 parameters:
    • Estimate of the total number of lines of new code to be developed
    • Estimate of the reuse cost based on an equivalent number of source lines of code (ESLOC), calculated using the reuse model.
    • An estimate of the number of lines of code that are likely to be modified because of changes to the system requirements.
  • Rate B based on five factors that you rate on a scale of 0 to 5, add the results of the factors and divide that by 100. Add that to 1.01.
    • Precedentedness
    • Development flexibility
    • Architecture/risk resolution
    • Team cohesion
    • Process maturity
  • Estimating calendar time required to complete a project:
    • TDEV = 3 * PM ^ (0.33 + 0.2 * (B - 1.01))
    • TDEV — nominal schedule for the project, in calendar months, ignoring any multiplier that is related to the schedule
    • PM — effort computed by the COCOMO model
    • B — complexity-related exponent
  • Projects are likely to run into significant problems if they try to develop software without allowing sufficient calendar time to complete the work.
