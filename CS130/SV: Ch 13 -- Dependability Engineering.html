Sommerville: Chapter 13 — Dependability Engineering
  • Three approaches to developing dependable software:
    • Fault avoidance.
    • Fault detection and correction.
    • Fault tolerance.
  • The cost of finding and removing the remaining faults in a software system rises exponentially as program faults are discovered and removed.
  • Do not remove faults if it is cheaper to pay for the consequences of failure than it would be to discover and remove the faults before system delivery.
  • The social and political acceptability of system failure must also be taken into account.

Redundancy and diversity
  • Fundamental strategies for enhancing the dependability of any type of system.
  • Redundancy means that spare capacity is included in a system that can be used if part of that system fails.
  • Diversity means that redundant components of the system are of different types, thus increasing the chances that they will not fail in unison.
  • Redundancy is great for high availability.
  • Diversity and redundancy make systems more complex and harder to understand.
  • Additional functionality must be introduced to switch from faulty to spare components

Dependable processes
  • The evidence that a dependable process has been used is often important in convincing regulators that the most effective software engineering practice has been applied in developing the software.
  • Data about process usage must be gathered to guarantee the process has been followed.
  • The process must be repeatable.
  • Examples of dependable processes:
    • Requirements reviews to check that the requirements are, as far as possible, complete and consistent.
    • Requirements management to ensure that changes to the requirements are controlled and that the impact of proposed requirements changes is understood by all developers affected by the change.
    • Formal specification.
    • System modeling, where the software design is explicitly documented as a set of graphical models.
    • Design and program inspections driven by checklists of common design and programming errors.
    • Static analysis, where automated checks are carried out on the source code of the program.
    • Test planning and management, where a comprehensive set of system tests if designed.

Dependable system architectures
Protection systems
  • A specialized system that is associated with some other system. 
  • Protection systems independently monitor their environment and, if the sensors indicate a problem that the controlled system is not dealing with, then the protection system is activated to shut down the process or equipment.
  • A protection system only includes the critical functionality that is required to move the system from a potentially unsafe state to a safe state.

Self-monitoring architectures
  • Perform computations on separate channels and compare the outputs of these computations.
  • To detect both HW and SW issues, they have to be designed so that:
    • The hardware used in each channel is diverse. In practice, this might mean that each channel uses a different processor type to carry out the required computations. This reduces the probability of common processor design faults affecting the computation.
    • The software used in each channel is diverse

N-version programming
  • Multiversion programming as a means to provide software redundancy and diversity.
  • Triple Modular Redundancy (TMR): hardware unit replicated three or more times. Result of each replication is sent to a voting system that chooses the final value.
  • Have number of teams independently implement a system to the same specification. Feed output from each program to a voting system.
  • Due to cost, only used if it is impractical to provide a protection system.

Software diversity
  • The ensure N-version success, you can
    • Include requirements for a choice of architecture for each team.
    • By requiring each team to implement solutions in different programming languages.
    • By requiring the use of different tools and development environments for the system.
    • By explicitly requiring different algorithms to be used in some parts of the implementation.
  • Teams often misunderstand the same part of the spec and make the same design and implementation errors because:
    • Members of different teams are often from the same cultural background and may have been educated using the same textbooks.
    • Incorrect requirements will propagate to all versions of the system.
  • Ofcourse, an N-version approach can be used for writing specifications as well.

Dependable programming
  • Control the visibility of information in a program. Only components that need to know something should know it. Information that is not directly relevant should be withheld.
  • Check all inputs for validity. Possible types of checks:
    • Range checks.
    • Size checks.
    • Representation checks.
    • Reasonableness check.
  • Provide a handler for all exceptions. What should exception handlers do?
    • Signal to a higher-level component that an exception has occurred, and provide data about the exception.
    • Carry out some alternative processing to that which was originally intended.
    • Pass control to a run-time support system that handles the exception. You should only use this approach when it is possible to move the system to a safe and quiescent state, before handing control to the run-time system.
  • Minimize use of error-prone constructs. Potentially error-prone construction include:
    • Unconditional branch (go-to) statements.
    • Floating-point numbers. Due to their inherent imprecision. Fixed-point numbers, with a pre-specified decimal precision are often more reliable.
    • Pointers.
    • Dynamic memory allocation. Due to memory leaks.
    • Parallelism.
    • Recursion.
    • Interrupts.
    • Inheritance. Because the code associated with an object may not all be in one place.
    • Aliasing. More than one name refers to the same entity in the program.
    • Unbounded arrays.
    • Default input processing.
  • Provide restart capabilities.
    • In a long transaction system, the time between starting to use the system and finishing work may be several minutes or hours. If the system fails during a long transaction, then all of the work may be lost.
    • In all these systems you should provide a restart capability that is based on keeping copies of data that is collected or generated during processing.
  • Check array bounds.
  • Include timeouts when calling external components.
  • Name all constants that represent real-world values.
